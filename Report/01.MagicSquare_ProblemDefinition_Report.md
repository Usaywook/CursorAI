## STEP 1 — 관찰

### Q1. 현재 우리가 해결하려는 상황은 무엇인가?

4×4 격자(grid)에 1부터 16까지의 서로 다른 정수가 배치되어 있다.
그런데 이 격자는 완성된 상태가 아니다 — 정확히 두 개의 칸이 비어 있다(값 `0`으로 표시).

우리가 마주한 상황:

> **"부분적으로 채워진 4×4 격자가 주어졌을 때,
> 비어 있는 두 칸에 어떤 숫자를 넣어야 모든 행·열·대각선의 합이 동일해지는가?"**

이것은 단순히 숫자를 채우는 문제가 아니다.
주어진 격자 상태를 **검증**하고, 빈 칸의 **위치를 탐색**하고, **두 가지 배치 조합을 시도**해 올바른 해를 찾아야 하는 탐색 + 검증의 복합 상황이다.

### Q2. 왜 4×4 마방진 문제를 다루는가?

| 관점 | 관찰 |
|---|---|
| **규칙의 명확성** | "모든 행·열·대각선의 합 = 34"라는 불변식이 수학적으로 정해져 있어 테스트 기준이 명확하다 |
| **입출력 계약의 단순함** | 입력(4×4 격자) → 출력(6개 정수: 좌표 2개 + 값 2개)으로 경계가 분명하다 |
| **복잡성의 적절함** | 너무 단순하지도, 너무 복잡하지도 않아 설계 연습에 적합하다 |
| **실패 시나리오의 다양성** | "해가 없는 경우", "입력 오류", "중복 숫자" 등 예외 상황이 풍부하다 |

마방진 자체가 목적이 아니라, 이 문제를 통해 소프트웨어 설계·테스트·검증 방법을 연습하는 것이 목적이다.

### Q3. 어떤 학습 또는 시스템 설계 맥락에서 등장하는가?

**① Cursor AI 보조 개발 학습**
요구사항을 어떻게 정의하고, AI에게 어떻게 전달하며, AI의 산출물을 어떻게 검증하는가?

**② Dual-Track TDD 실습**
도메인 로직(Logic Track)과 UI 계약(UI Contract Track)을 분리해서
실패 테스트부터 작성하고, 최소 구현 후 리팩터링하는 사이클을 익힌다.

**③ ECB 아키텍처 설계 연습**
Entity(도메인 규칙), Control(유스케이스 조합), Boundary(입출력 처리)로
책임을 어떻게 나누는지를 이 문제를 통해 체득한다.

## STEP 2 — Why #1

### Q: 왜 마방진을 완성해야 하는가?

**가정 답변 A:**
> "사용자가 입력한 부분 격자가 올바른 마방진이 되는지 확인하고,
> 빈 칸에 들어갈 정답을 알려줘야 하기 때문이다."

### 드러나는 구조적 문제

답변 A를 분해하면 세 가지 서로 다른 행위가 뒤섞여 있다.

| 행위 | 내용 | 숨겨진 문제 |
|---|---|---|
| **① 검증** | "올바른 마방진인지 확인한다" | 무엇을 기준으로 올바름을 판단하는가? |
| **② 탐색** | "빈 칸의 위치를 찾는다" | 빈 칸이 0개, 1개, 3개 이상이면 어떻게 처리하는가? |
| **③ 계산** | "정답을 알려준다" | 해가 유일한가? 해가 없다면? |

**핵심 불편함:**
> "완성한다"는 단어 하나에 검증·탐색·계산이 모두 압축되어 있어,
> 각 책임의 경계가 명확하지 않다.

구체적 불편함:
- **불편함 ①** 검증 기준이 암묵적 (행 합만? 열과 대각선도?)
- **불편함 ②** 빈 칸 개수 가변적 (입력 검증인지 도메인 규칙인지 경계 모호)
- **불편함 ③** "정답을 알려준다"의 형식 불분명 (좌표+값? 완성 격자?)

## STEP 3 — Why #2

### Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?

#### ① 반복 가능성 (Repeatability)

손으로 계산하면 격자가 바뀔 때마다 처음부터 다시 해야 한다.

```
손 계산:  격자 변경 → 처음부터 다시
프로그램: 격자 변경 → 동일 함수 재호출 → 즉시 결과
```

#### ② 검증 자동화 (Automated Verification)

마방진의 올바름을 사람이 확인하려면 4행 + 4열 + 2대각선 = **총 10번의 덧셈**을 매번 손으로 해야 한다.
프로그램은 이 10번의 검증을 한 번의 함수 호출로 끝낸다.

#### ③ 오류 방지 (Error Prevention)

| 오류 유형 | 예시 | 프로그램에서의 처리 |
|---|---|---|
| **입력 오류** | 17 입력, 중복 숫자, 빈 칸 3개 | 입력 검증 단계에서 즉시 거부 |
| **계산 오류** | 덧셈 실수, 조합 누락 | 알고리즘이 모든 조합을 빠짐없이 시도 |

#### ④ 규칙 기반 사고 훈련 (Rule-based Thinking)

| | |
|---|---|
| 모호한 직관 | **→** 명확한 규칙 |
| "대충 맞는 것 같다" | **→** "모든 행·열·대각선의 합 = 34, 단 1~16 중복 없음" |
| 암묵적 가정 | **→** 명시적 불변식 |
| "당연히 빈 칸은 2개" | **→** I1: 빈 칸은 정확히 2개여야 한다 |
| 결과 중심 사고 | **→** 과정 중심 설계 |
| "답이 8과 1이다" | **→** "왜 8이 먼저인가? Step A인가 Step B인가?" |

프로그램 구현은 **사고의 허점을 코드가 드러내게 만드는** 훈련이다.

## STEP 4 — Why #3 (핵심 발견)

### Q: 왜 이 문제를 TDD 방식으로 설계하려 하는가?

#### 무엇이 통제되어야 하는가?

마방진 프로그램에는 실패할 수 있는 지점이 4곳 있다.

```
[입력] → [검증] → [탐색] → [계산] → [출력]
          ↑          ↑         ↑         ↑
        오류 ①     오류 ②    오류 ③    오류 ④
```

| 오류 지점 | 실패 예시 | 통제 방법 |
|---|---|---|
| ① 검증 | 빈 칸이 3개인 격자를 통과 | 입력 검증 테스트 |
| ② 탐색 | 빈 칸 좌표를 잘못 찾음 (0-index vs 1-index 혼용) | 탐색 함수 단위 테스트 |
| ③ 계산 | Step A만 시도하고 Step B를 시도하지 않음 | 해 계산 테스트 (두 조합 모두 검증) |
| ④ 출력 | 좌표와 값의 순서가 뒤바뀜 | 출력 계약 테스트 |

> TDD는 이 4곳을 **각각 독립적으로 분리해서 잠그는** 방법이다.

#### 어떤 불변 조건이 존재하는가?

불변식은 발견하는 것이 아니라 **결정하는 것**이다.

> "합이 34여야 한다"는 수학이지만,
> "빈 칸이 정확히 2개여야 한다"는 **설계 결정**이다.
> **TDD는 이 결정을 테스트 코드에 박제한다.**

#### 왜 입력/출력이 명확해야 하는가?

입출력이 모호하면 테스트를 쓸 수 없다. 테스트를 쓸 수 없으면 구현이 맞는지 틀린지 알 수 없다.

```
입력이 명확해야  → Boundary(입력 검증)가 무엇을 막아야 할지 안다
출력이 명확해야  → Control(계산)이 무엇을 반환해야 할지 안다
둘이 명확해야    → Entity(도메인 규칙)가 어디까지 책임지는지 안다
```

## STEP 5 — 진짜 문제 정의

### 1. 표면 문제 정의 (잘못된 정의)

> "4×4 마방진에서 비어 있는 두 칸을 찾아서 채워 넣는 프로그램을 만든다."

이 정의가 잘못된 이유:

| 오류 | 설명 |
|---|---|
| **결과 중심** | "채워 넣는다"는 행위만 말하고, 무엇이 올바름인지를 말하지 않는다 |
| **전제 숨김** | "비어 있는 두 칸"이라는 조건이 항상 성립한다고 가정하지만 검증하지 않는다 |
| **경계 없음** | 입력이 잘못되었을 때, 해가 없을 때 어떻게 되는지 정의하지 않는다 |
| **책임 뭉침** | 검증·탐색·판단·전달이 "채워 넣는다" 한 문장에 압축되어 있다 |

### 2. 개선된 문제 정의 (정확한 정의)

> **"부분적으로 채워진 4×4 격자를 입력받아,**
> **정해진 조건을 만족하는지 검증한 뒤,**
> **두 개의 빈 칸에 들어갈 값을 결정하여,**
> **그 위치와 값을 정해진 형식으로 응답하는 시스템을 정의한다."**

| 항목 | 표면 정의 | 개선된 정의 |
|---|---|---|
| **주어** | 프로그램이 "채운다" | 시스템이 "검증하고, 결정하고, 응답한다" |
| **전제 조건** | 암묵적 | 명시적 ("정해진 조건을 만족하는지 검증") |
| **책임** | 하나의 행위로 뭉쳐 있음 | 검증 → 결정 → 응답으로 분리됨 |
| **실패 처리** | 언급 없음 | "검증 실패" 경우가 존재함을 내포 |
| **출력** | "채워 넣는다" (형식 미정) | "위치와 값을 정해진 형식으로 응답" |

### 3. 핵심 Invariant (불변식 목록)

#### 입력 불변식

| ID | 불변식 | 위반 시 의미 |
|---|---|---|
| **I1** | 격자는 반드시 4행 4열이어야 한다 | 격자 구조 자체가 성립하지 않음 |
| **I2** | 빈 칸은 정확히 2개이어야 한다 | 탐색과 결정의 전제가 무너짐 |
| **I3** | 빈 칸이 아닌 값은 1 이상 16 이하이어야 한다 | 마방진의 수학적 조건이 성립 불가 |
| **I4** | 빈 칸이 아닌 값 사이에 중복이 없어야 한다 | 해가 존재할 수 없음 |

#### 출력 불변식

| ID | 불변식 | 위반 시 의미 |
|---|---|---|
| **I5** | 응답은 6개의 정수 — [r1, c1, v1, r2, c2, v2] | 응답의 형식이 무너지면 수신자가 해석 불가 |
| **I6** | 위치는 1부터 시작하는 번호로 표현한다 | 내부(0-index)와 외부(1-index) 간 해석 불일치 |
| **I7** | 완성 격자의 모든 행·열·대각선 합은 34이어야 한다 | 마방진 조건을 만족하지 않는 답은 답이 아님 |

#### 해 결정 불변식

| ID | 불변식 | 위반 시 의미 |
|---|---|---|
| **I8** | 두 빈 칸에 넣을 값은 격자에 아직 없는 숫자 중에서만 선택된다 | 중복 발생 |
| **I9** | 두 값의 배치 순서는 두 가지 경우를 모두 시도한 뒤 성립하는 것을 선택한다 | 한 가지만 시도하면 실패로 오판할 수 있음 |
| **I10** | 어떤 배치도 마방진 조건을 만족하지 않으면, 해가 없음을 명시적으로 알려야 한다 | 오답을 정답으로 반환하거나 침묵하는 것은 더 큰 오류 |

### 4. 우리가 실제로 훈련하려는 사고 능력

| 사고 능력 | 설명 |
|---|---|
| **① 전제를 발견하는 능력** | "이것은 항상 성립한다"고 가정하고 있는 것이 무엇인가를 먼저 묻는 능력 |
| **② 책임을 분리하는 능력** | "채워 넣는다"를 검증 / 탐색 / 결정 / 응답으로 쪼갤 수 있는 능력 |
| **③ 실패를 먼저 상상하는 능력** | 정상 경로보다 비정상 경로를 먼저 정의하는 능력 |
| **④ 계약을 명문화하는 능력** | 입출력 형식을 "그냥 안다"가 아니라 검증 가능한 명제로 남기는 능력 |
| **⑤ 관찰에서 정의로 이행하는 능력** | "마방진이 뭔지 안다"에서 "무엇을 보장해야 하는가"로 스스로 이행하는 능력 |

## 최종 문제 정의문

> **이 시스템은 부분적으로 채워진 4×4 격자를 입력받아,**
> **I1~I4의 입력 불변식을 검증하고,**
> **I8~I9의 해 결정 원칙에 따라 두 빈 칸의 값을 결정하며,**
> **I5~I7의 출력 불변식을 만족하는 형식으로 응답한다.**
> **어떤 불변식도 위반될 경우, 시스템은 그 이유를 명시적으로 알린다.**

## Why Chain 누적 요약

```
Why #1: 마방진을 완성해야 하는가?
  → 검증·탐색·계산·출력 4가지 책임이 필요하기 때문
  → ⚠ 4가지 책임이 뭉쳐 있음 → 경계 분리 필요

Why #2: 왜 프로그램으로 구현하는가?
  → 반복 가능성 + 검증 자동화 + 오류 방지 + 규칙 강제 번역
  → ⚠ 규칙이 불변식으로 정의되지 않으면 코드로 표현 불가

Why #3: 왜 TDD 방식인가? (핵심 발견)
  → 4곳의 오류 지점을 독립 통제
  → 10개 불변식을 테스트로 박제
  → 입출력 계약이 레이어 분리의 전제조건
  → ✅ 결론: 입출력 계약과 불변식이 먼저 정의되어야 구현이 가능하다
```
